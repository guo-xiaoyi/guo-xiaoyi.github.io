How to use oTree + Prolific for a longitudinal study with cumulative progress

January 2026 | Methods note | 9 min read

This is a practical blueprint for running repeated waves on Prolific while keeping participant progress and data consistent inside oTree. The goal is simple: recruit once, return the same people, and keep their progress intact without manual fixes.

Why this setup works

Longitudinal online experiments are fragile. You need stable identifiers, clean wave scheduling, and a predictable participant flow. oTree is great for structured tasks, while Prolific is best for recruitment and reminders. Together, they can run a study with multiple waves as long as you establish one persistent key and a simple rule for progress.

I assume you will invite the same pool back for each wave. If you are using new participants each wave, drop the progress logic and keep only the Prolific integration.

Core architecture

Think in three stable layers:

  1. Prolific provides the participant ID and handles recruitment and re-contact.
  2. oTree stores a permanent key and wave status in participant.vars.
  3. Your data merge uses the Prolific ID plus the oTree participant code.
The minimal persistent key is PROLIFIC_PID. Keep it in every wave. Every other field can be reconstructed if you have that key and the timestamps.

Wave planning

Decide three things before you code:

  • How many waves and how far apart (for example, 4 waves, 2 weeks apart).
  • How you will handle late returners (allow a grace period or close the wave).
  • What constitutes progress (for example, completed wave count or task score).

oTree session config

Use one session config per wave, and pass a wave integer. This keeps your data simple and your export tidy. 

# settings.py
SESSION_CONFIGS = [
    dict(
        name="longi_wave_1",
        display_name="Longitudinal Wave 1",
        app_sequence=["intro", "task", "survey"],
        num_demo_participants=5,
        wave=1,
    ),
    dict(
        name="longi_wave_2",
        display_name="Longitudinal Wave 2",
        app_sequence=["intro", "task", "survey"],
        num_demo_participants=5,
        wave=2,
    ),
]

Capture the Prolific ID once

Prolific sends PROLIFIC_PID in the URL. In oTree, it appears as participant.label. Save it to participant.vars so it persists. 

# intro/pages.py
class Intro(Page):
    def vars_for_template(self):
        pid = self.participant.label
        if pid:
            self.participant.vars["prolific_pid"] = pid
        return {"prolific_pid": self.participant.vars.get("prolific_pid")}

This makes the ID available across apps and across waves.

Track cumulative progress

Use a single integer that updates at the end of each wave. This is the simplest durable signal that a participant has completed the wave. 

# survey/pages.py
class Results(Page):
    def before_next_page(self):
        completed = self.participant.vars.get("waves_completed", 0)
        self.participant.vars["waves_completed"] = completed + 1

If you also need performance scores, store them in participant.vars as a list keyed by wave.

Participant flow on Prolific

  • Wave 1: recruit with a standard study and save the Prolific IDs.
  • Wave 2+: use a custom allowlist, inviting only IDs from Wave 1.
  • Keep a fixed completion code, and store it in the oTree end page.

This ensures that each wave only admits eligible participants and makes it easy to calculate attrition.

Data merge and QA

Merge on both prolific_pid and participant_code. This avoids duplicate rows if a participant starts the same wave twice.

  • Export oTree wide and long formats after each wave.
  • Keep a simple tracking sheet for invites, starts, and completions.
  • Flag any participant with multiple starts in the same wave.

Common pitfalls

  • Not saving the Prolific ID early enough, which breaks the panel link later.
  • Changing app sequences between waves and creating mismatched data fields.
  • Letting a wave remain open while sending new invites, which confuses returners.
  • Relying on completion codes as keys. They are not unique.

Minimal checklist

  • Store PROLIFIC_PID in participant.vars on the first page.
  • Define a wave number in the session config.
  • Update waves_completed at the end of each wave.
  • Invite the same IDs through a Prolific allowlist.
  • Merge by both IDs and save wave timestamps.
